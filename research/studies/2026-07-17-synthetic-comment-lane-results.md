# 2026-07-17 合成コメント注入・通常 naka レーン観測結果

## 目的

記録済みの公式視聴ページを外部通信なしで再生し、`nvComment` 応答のコメント配列だけを合成入力へ置換して、公式描画コードの処理順、レーン座標、同一レーン再利用境界を測る。

この実験は、既存動画の見た目へ合わせる校正ではなく、入力と出力を制御したブラックボックス観測から一般規則を反証可能な形で得ることを目的とする。

## 実行条件

- アーカイブ: `research/captures/sm6240144-baseline-20260717/manifest.json`（Git管理外）
- ブラウザー: Google Chrome `150.0.7871.116`
- 外部通信: ローカルプロキシ、DNS失敗規則、CDP Fetch遮断により禁止
- コメント応答: 記録済みthread構造を維持し、指定forkのコメント配列だけを置換
- 観測: ページスクリプト実行前にCanvas 2Dの `fillText`、`strokeText`、`drawImage` をhook
- 時刻源: ページ上のvideo要素へ `currentTime` を設定し、seek関連イベントを発火
- ローカル成果物: `research/runs/`（Git管理外）

ベースライン取得環境では、NicoCacheが `www.nicovideo.jp/local/` 配下の非公式資産を9件挿入していた。その中の `CommentFilter2` が空の許可リストで全コメントコマンドを削除することを確認したため、隔離再生ではこれらを無効化している。無効化前のコマンド観測は採用せず、通常レーンの主要結果も無効化後に再測定した。

Chrome 150のGPUプロセスは、管理実行環境で `GPUPersistentCache/DawnGraphiteCache` を開く際にアクセス拒否となることがあった。このため、オフライン再生だけにChromiumのテスト用 `--disable-gpu-sandbox` を指定した。レンダラープロセスのサンドボックスと外部通信遮断は維持し、オンラインキャプチャには指定しない。スイッチの用途は[Chromiumソース](https://chromium.googlesource.com/chromium/src/+/HEAD/content/public/common/content_switches.cc)で確認した。

## 基盤成立の確認

次の合成コメントが公式コードのCanvas描画へ到達した。

- `vposMs = 0` の1コメント
- `vposMs = 5000` の1コメントを5秒seek後に描画
- 同一 `vposMs` の4コメント

合成本文は公式Canvas上の `strokeText` と `fillText`、外側Canvasへの `drawImage` の両方で一致した。したがって、動画メディア本文をローカルfixtureへ置換しなくても、静的な時刻プローブにはnative videoの `currentTime` を利用できる。

## 同一時刻の処理順

同一 `vposMs` の4コメントで次を比較した。

1. 応答配列と `no` がともに昇順
2. 応答配列だけを逆順
3. 応答配列は維持し、`no` だけを逆順

公式コードは全ケースで `no` 昇順に処理した。応答配列順は処理順を決めない。

## レーン座標

同時刻4コメントの外側Canvasに対する `drawImage` の変換行列から、Y平行移動成分を取得した。

| 処理順 |             Y |
| -----: | ------------: |
|      1 |   1.560843 px |
|      2 |  69.725441 px |
|      3 | 137.890045 px |
|      4 | 206.054642 px |

隣接ピッチの中央値は `68.164598 px` だった。`drawImage` の引数に現れる `destinationY = -27` はsource canvas内のpaddingであり、レーンYではない。

GPUキャッシュとGraphiteを一時的に無効化した観測、および公式既定のGraphiteへ戻した観測で、4つのY、ピッチ、処理順は完全一致した。

NicoCacheの `/local/` 注入を無効化した公式経路でも、4つのY、ピッチ、処理順は完全一致した。

## 同幅2コメントの同一レーン再利用境界

本文が同幅になる2コメントを、先行Aと後続Bとして投入した。両方のsource canvasは `546 x 120 px` だった。

|  時刻差 |        AのX |     BのX | レーン結果 |
| ------: | ----------: | -------: | ---------- |
| 1287 ms | 327.6575 px | 813.5 px | 別レーン   |
| 1288 ms | 327.2800 px | 813.5 px | 同一レーン |

1281〜1287 msは別レーン、1288〜1300 msは同一レーンだった。境界の2値は各3回再測定し、1287 msは3回とも別レーン、1288 msは3回とも同一レーンになった。公式既定のGraphiteへ戻した追加測定でも同じ結果だった。

X差から同幅コメントの速度を求めると約 `377.5 px/s` になる。公式source canvasの左右padding合計を従来観測どおり約 `59.6 px` とすると、本文幅は約 `486.4 px` である。

```text
486.4 px / 377.5 px/s ≒ 1.2885 s
```

実測境界 `1288 ms` とほぼ一致するため、この同幅・通常 `naka` ケースは次の規則で説明できる。

```text
same lane is allowed iff:
  dt >= inkWidth / speed
```

### 幅を変えた反証

Canvas観測器で `measureText()` の値も描画時に記録し、内部scale `2` を適用した文字幅と境界を比較した。

| 本文 | source幅 | 実測文字幅 |        速度 | 文字幅 / 速度 | 別レーン最終値 | 同一レーン開始値 |
| ---- | -------: | ---------: | ----------: | ------------: | -------------: | ---------------: |
| 短文 |   276 px |  216.08 px | 310.50 px/s |     695.91 ms |         702 ms |           703 ms |
| 中間 |   546 px |  485.26 px | 377.50 px/s |    1285.46 ms |        1287 ms |          1288 ms |
| 長文 |  2466 px | 2404.79 px | 857.50 px/s |    2804.42 ms |        2805 ms |          2806 ms |

実測境界と計算値の差は `1.58〜7.09 ms` だった。source canvas全幅を使う計算より、Canvasが実測した文字幅を使う計算の方が境界を説明する。数msの差が、時刻量子化、stroke幅、速度または座標の丸めのどれに由来するかは未確定である。

さらに短文と長文を前後で入れ替えた。

| 先行 | 後続 | 2805 ms  | 2806 ms    |
| ---- | ---- | -------- | ---------- |
| 短文 | 長文 | 別レーン | 同一レーン |
| 長文 | 短文 | 別レーン | 同一レーン |

両方向とも長文同幅ペアの境界に一致した。したがって、過去のオンラインCDP観測で得ていた次の仮説は、今回の隔離再生でも支持された。

```text
same lane is allowed iff:
  dt >= max(
    inkWidth(first) / speed(first),
    inkWidth(second) / speed(second)
  )
```

短・中・長の境界2値と、短→長・長→短の境界2値は、`/local/` 注入を無効化した公式経路でもすべて同じ結果になった。

## コマンド経路の成立確認

`/local/` 注入を無効化した後、同一合成入力の `position=ue`、`color=red`、`size=big`、`isPremium=true` を公式コードが次のように描画した。

- fill color: `#ff0000`
- font: `600 39px Arial, ...`
- source canvas: `786 x 174 px`
- X: 2コメントとも `328 px`で固定
- Y: `2.135551 px`、`100.797089 px`

位置、色、サイズがすべて反映されたため、合成応答からコマンド規則を研究できる状態になった。

### サイズ別の単行レーン

`naka`、`ue`、`shita` の各位置へ同一時刻2コメントを投入した。単行のfont、source canvas高さ、隣接Yピッチは位置によらず同じだった。

| size   | font size | source高さ | 隣接Yピッチ |
| ------ | --------: | ---------: | ----------: |
| small  |     18 px |      84 px |    46.47 px |
| medium |     27 px |     120 px |    68.16 px |
| big    |     39 px |     174 px |    98.66 px |

上固定は上から下へ、下固定は下から上へ同じ絶対ピッチで積まれた。

| size   | `ue` 1本目Y | `ue` 2本目Y | `shita` 1本目Y | `shita` 2本目Y |
| ------ | ----------: | ----------: | -------------: | -------------: |
| small  |    -0.51 px |    45.96 px |      721.23 px |      674.76 px |
| medium |     1.56 px |    69.73 px |      701.60 px |      633.43 px |
| big    |     2.14 px |   100.80 px |      671.67 px |      573.01 px |

`naka` も各サイズで `ue` と同じ上側Yとピッチを使った。少なくとも単行・同サイズでは、位置ごとに別の縦寸法を持たず、サイズから得た共通ピッチを積み方向だけ変えて使用している。

### サイズ別の通常 naka 再利用境界

同じ本文を `small`、`medium`、`big` で描画し、レーン0を再利用する最小時刻差を測った。

| size   | source幅 | 実測文字幅 |        速度 | 文字幅 / 速度 | 別レーン最終値 | 同一レーン開始値 |
| ------ | -------: | ---------: | ----------: | ------------: | -------------: | ---------------: |
| small  |   366 px |  323.51 px | 337.00 px/s |     959.96 ms |         961 ms |           962 ms |
| medium |   546 px |  485.26 px | 377.50 px/s |    1285.46 ms |        1287 ms |          1288 ms |
| big    |   786 px |  700.93 px | 431.50 px/s |    1624.41 ms |        1626 ms |          1627 ms |

全サイズで実測境界と文字幅通過時間の差は `2.04〜2.59 ms` に収まった。したがって、サイズコマンドはfont、source寸法、速度を変えるが、通常 `naka` の同一レーン衝突判定式自体は変えない。

## 現時点で支持される規則

1. 通常 `naka` は同一時刻内で `no` 昇順に処理される。
2. 通常状態では上から等間隔のレーンを使う。
3. 2コメントのレーン0再利用境界は、source canvas全幅ではなくCanvasが測った文字幅の通過時間とほぼ一致する。
4. 幅が異なる場合は、先行・後続の向きによらず、両コメントの文字幅通過時間の長い側が境界を決める。
5. 単行コメントのfont sizeは `small=18 px`、`medium=27 px`、`big=39 px`で、隣接Yピッチは位置によらずサイズごとに共通である。
6. `ue` は上から下、`shita` は下から上へ同じ絶対ピッチで積む。
7. `small` / `medium` / `big` は通常 `naka` の速度と文字幅を変えるが、同一レーン再利用には共通の `inkWidth / speed` 規則を使う。

## 適用範囲と未確定事項

通常レーン再利用について確定したのは、単行・通常 `naka`、2コメント、通常レーン範囲内である。固定位置とサイズについては同時2コメントまで確認した。次は次の順に直交プローブを追加する。

複数行の結果は [複数行コメント・可変高スロット観測](./2026-07-17-multiline-slot-results.md) に分離した。次は次の順で直交プローブを追加する。

1. 数msの境界差が時刻量子化、stroke幅、または丸めのどれで生じるか分離する。
2. `full`、フォント、色が処理順や予約幅へ影響するか分離する。
3. 通常表示高を超える同時投入でoverflow規則と非決定性を再検証する。
4. 成立した規則を未使用本文・未使用幅へホールドアウトし、成立したものだけを `src/` へ移す。
