# 2026-07-17 固定コメント未観測範囲の検証結果

## 結論

固定コメントの縦配置は、作品、`ca`、ユーザー、forkなどへの個別分岐ではなく、表示中の予約区間と、横幅・行数の調整後に確定する論理slot高で説明できる。

1. 上固定は上端から、下固定は下端から、既存予約と重ならない最初の区間へ置く。
2. 新しいslot高が表示高未満で空き区間だけが足りない場合は、表示範囲内の一様ランダムYへ1回だけ退避する。衝突時の再抽選はしない。
3. ランダム退避したコメントも予約として後続を遮る。後続が入る穴があれば穴を使い、なければ後続も独立にランダム退避する。
4. 新しいslot高が表示高以上の場合は探索せず、上固定は上端、下固定は `表示高 - slot高` へ置く。後続レイヤーも同じ座標になる。
5. `ue`、`shita`、`naka` は互いに独立した予約系を持つ。
6. 固定表示時間は3000msで、`+2999ms`までは同時表示、`+3000ms`で先行コメントが失効する。seek後も生存予約のYは変えず、失効して空いた端は新しいコメントが使う。
7. 横幅調整は文字数ではなく実測文字幅に依存し、その結果のフォントサイズと描画scaleが縦slot高にも反映される。

第5話EDの矩形、白抜き、歌詞は16行の `ue big full mincho` レイヤーで、横幅調整後も論理slot高が768以上になる。したがって全レイヤーが同じ上端へ重なり、作品固有・`ca`固有の分岐は不要である。

## 実験方法

- 入力生成: `research/tools/run-fixed-comment-matrix.mjs`
- 教師: 保存済み公式プレイヤー資産の外部通信遮断リプレイ
- 観測: source canvas寸法、outer canvasの変換行列、処理sequence、video時刻
- 生データ: `research/runs/fixed-comment-matrix-*`（Git管理外）
- 個別レイヤー対照: `research/scenarios/fixed-full-mincho-layered-ca.json`

20 profile、合計310ケースを観測した。同一条件を別Chromeプロセスでも反復し、決定的な端固定とランダム退避を区別した。直交する軸を先に単独観測し、境界付近、機能の交差、順序反転、seek、異なるidentityとforkを反証ケースに使った。

| 条件       |           Y1 |      Y2 | 分類                         |
| ---------- | -----------: | ------: | ---------------------------- |
| big 1行    |       -0.674 |  97.988 | 整列                         |
| big 15行   | 表示上端付近 | run依存 | 空き不足のランダム退避       |
| big 16行   |       -1.340 |  -1.340 | 表示高以上の端重畳           |

## 横幅と行数の一般規則

公式source座標で、通常の固定コメントがまず目標とする最大幅はplainが約512、`full`が約682である。現在の768基準レイアウトではそれぞれ表示幅の `0.75` 倍と `1.0` 倍に相当する。

1. `ender`以外の複数行は、先に行数境界の縦縮小を適用する。
2. その状態の最大行幅が狭い目標幅を超えた通常複数行コメントは、元のサイズの行メトリクスへ戻し、2倍の目標幅（plain約1024、`full`約1364）へ再フィットする。
3. `ender`は行数縮小を行わず、狭い目標幅へ直接フィットする。
4. フォントを公式source 10px相当まで縮めても収まらない場合は、描画全体を0.1刻みで縮小する。
5. 最終フォント、行送り、描画scaleからslot高を求め、そのslot高だけを配置器へ渡す。

big/mincho単行では、10文字まではsource font 39px・縦間隔98.662pxだった。30文字では17px・43.287px、60文字では10pxに0.8 scale・21.744px、120文字では10pxに0.4 scale・10.972pxへ縮小した。30文字の`full`は22px・57.594pxであり、`full`が単なる配置フラグではなく幅上限を変えることも確認した。

複数行には不連続な切替がある。small 7行は50文字でsource 10pxのままだが55文字で17px経路へ、medium 5行は35文字の14pxから40文字の24pxへ、big 3行は25文字の20pxから26文字の37pxへ切り替わった。この不連続は「縦縮小後に狭い幅を超えたら元サイズ・広い幅へ戻す」で一貫して説明できる。

ASCIIの広い字と細い字、emoji、結合文字、混在文字、ideographic/em spaceでも、文字数ではなくCanvasの実測幅で同じ規則になった。tabは公式側でem space 2個へ正規化される。

## fallbackと予約の補足

19行smallを20回独立再生した上固定fallbackの正規化Yは、最小0.001、最大0.976、平均0.475、四分位binが5/5/6/4だった。一様乱数と矛盾せず、固定レーン番号やuserId由来の決定値では説明できない。

ランダムYへ置かれた予約も通常予約と同じく後続探索へ参加する。残った穴に後続の小さいslotが収まればそこへ置き、収まらなければ後続も独立に1回だけランダム配置する。既存予約と衝突しない位置が出るまで再抽選する挙動は観測されなかった。

`overlay-tests/fixtures/so31723295-ed-comments.json` から `leaf` かつ `ue big full mincho` の層だけを抽出し、規則導出には使わず最終テストへ回した。

- 実装の`Comment.prepare()`を通した全層の768基準slot高が768以上
- 全バッチの全レイヤーがY=0へ解決

検証は `research/tools/tests/fixed-ca-holdout.test.ts` に固定した。画素単位一致は判定条件にしていない。公式側でもロードや再生位置によりレーン選択が変わるため、視覚確認では矩形、白抜き、歌詞の相対的な重なりが保持されることを確認対象とする。

## 実行環境の切り分け

当初interaction profileが停止したように見えた原因は公式処理ではなく、管理実行環境がChromeのCDP domain有効化を遮断していたことだった。`progress.json`を追加して停止箇所を特定し、許可されたChrome実行で混在4ケースを完走した。得られなかった座標を推測で補ったケースはない。

追加で設計した「極端な横幅×`full`/`ender`」11ケースは、実行枠上限によりChrome起動前に拒否された。この交差の各構成要素は既に別profileで観測済みであり、実装では同じ純粋関数を通す交差テストを置いた。実行可能になった時に `width-extreme-features` profileで追試できるよう入力は保存している。

## `src/` への反映

幅調整と固定配置を別の純粋な決定として実装した。

```text
metrics = fitMeasuredWidth(originalMetrics, verticallyReducedMetrics, full, ender)
slotHeight = metrics.lineAdvance * lineCount * metrics.drawScale

if slotHeight >= displayHeight:
    y = 0                         # ue
    y = displayHeight-slotHeight  # shita
else if edgeから探索した空き区間がある:
    y = 最初の空き区間
else:
    y = random() * (displayHeight-slotHeight)
```

通常整列、穴の再利用、fallback予約、表示高以上、期限切れ、seek、文字幅、行数と幅の不連続、`full`/`ender`、実CA全11バッチを自動テストで覆った。固定コメントを画面外へ無制限に積み上げる旧処理と、特定フォント・単行幅だけを補正する旧ヒューリスティックは除去した。
