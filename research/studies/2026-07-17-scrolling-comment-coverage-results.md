# 2026-07-17 スクロールコメント未観測範囲の検証結果

## 結論

保存済み公式プレイヤーに対するスクロールコメントの幾何、速度、レーン再利用、開始・終了、seek、メタデータ、文字種の未観測範囲を、12 profile・431ケースで検証した。固定コメント321ケースと合わせた直交表は752ケースである。

スクロールコメントはコメントアート、`ca`、`full`、投稿者、fork、作品ごとの分岐を必要とせず、次の一般規則で説明できる。

1. sizeと行数から文字サイズ・行送り・slot高を決める。`ender`だけが行数起因の縮小を止める。
2. 最大行のCanvas実測幅をink幅として使い、横幅に応じたフォント縮小や描画scaleは行わない。
3. ink幅が公式の描画領域を4秒で横切るように速度を決める。
4. コメント時刻の2000ms前に生成し、1000ms前にテクスチャ左端が描画領域右端へ到達する。
5. 同方向の同一レーンは、先行コメントのink幅が開始点を通過した時点から再利用できる。
6. テクスチャが外側Canvasを完全に抜けるまで描画し、固定の失効時刻では消さない。
7. `full`と投稿メタデータはスクロールの寸法、速度、位置、予約に影響しない。

## 数式

外側Canvas幅を `W`、公式描画領域幅を `R`、最大行の実測ink幅を `M`、inkの片側テクスチャ余白を `P` とする。

```text
R = W * 1024 / 1364
P = fontSize / 2 + 3
speed = (R + M) / 4000               # px/ms
activation = vpos - 2000
laneReuse = ceil(M / speed)           # ms
```

右から左の場合、activation時のテクスチャ左端は `renderRight + speed * 1000` である。したがってvposの1000ms前に描画領域へ入り始め、そこからink右端が描画領域左端を抜けるまでが4000msになる。破棄はテクスチャ右端が外側Canvas左端を越えた後に行う。境界フレームでは描画後に破棄されるため、独立seekの観測には1フレーム程度の揺れがある。

## 実験方法

- 入力生成: `research/tools/run-fixed-comment-matrix.mjs`
- 教師: 保存済み公式プレイヤー資産の外部通信遮断リプレイ
- 観測: source canvas寸法、実測文字幅、変換行列、video時刻、レーンY
- 生データ: `research/runs/comment-matrix-scroll-*`（Git管理外）
- 検証profile: `scroll-features`、`scroll-width`、`scroll-sync`、`scroll-metadata`、`scroll-viewport`、`scroll-seek`、`scroll-lifecycle`、`scroll-lifecycle-boundary`、`scroll-lifecycle-edge`、`scroll-reuse-boundary`、`scroll-glyph`、`scroll-cleanup-boundary`

各離散入力は同値類へ分割し、連続量は切替点の直前・一致・直後を独立プロセスで測った。単独軸の観測後に、size・行数・font・`full`・`ender`・幅を交差させ、規則導出に使っていない境界ケースで反証した。

| ------------ | ------------------------------------------------------------- | ------------------------------------------------------ |

## 代表値

1364幅では描画領域幅が1024となる。big/mincho単行の実測値は次のとおりだった。

|  ink幅 |       speed | 同一レーンを初めて再利用できる整数時刻差 |
| -----: | ----------: | ---------------------------------------: |
|   78px | 0.2755px/ms |                                    284ms |
| 2340px |  0.841px/ms |                                   2783ms |
| 9360px |  2.596px/ms |                                   3606ms |

複数行の自動縮小後も同じ式となり、small 7行・30文字は600pxと約0.406px/msで1478ms、medium 5行は840pxと約0.466px/msで1803ms、big 3行は1200pxと0.556px/msで2159msだった。

`full`の有無を含む300件以上の対照で、source font、source/変換後寸法、X、Yに差はなかった。同一vposに`full mincho ender`を置いた場合だけ他レイヤーを変形する旧CO分岐は、公式側に対応する挙動がなく、特定CAへの過適合だった。

## 網羅性の判定

無限個の文字列や実数時刻を逐一列挙する代わりに、結果を変え得る入力を有限の同値類へ分けた。離散コマンド、size、font、行数縮小境界、文字幅の代表と極値、時刻境界、同時刻順序、レーン衝突境界、viewport、seek、identity、forkを観測し、交差ケースと未使用境界で反証した。残った連続量はCanvas実測幅と線形式へ還元され、ASCII・emoji・結合文字を含む追加ケースでも反例が出なかった。

このため、保存済み公式バンドルが受理する描画入力に対する一般規則の探索は完了と判定する。将来の公式バンドル変更、OSのフォント差、ブラウザのCanvas実装差まで永久に証明するものではない。画素単位一致は目標にせず、公式自身に揺れがあるレーン選択と境界フレームは同値な配置クラスとして扱う。

## `src/` への反映

速度と寿命を `src/comment/nico-scroll.ts` の純粋関数へ集約し、固定コメントの幅フィッターと完全に分離した。`full`専用速度、幅bucket、特定幅scale、同一vposの`full mincho ender`連動、`full`の先頭レーン固定を削除した。スクロールのデフォルト設定は公式式を使う`null`とし、明示的な`scrollVisibleDurationMs`だけを4秒の横断時間に対する上書きとして扱う。
