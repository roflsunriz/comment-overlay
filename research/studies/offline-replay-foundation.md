# オフライン再生基盤: 研究計画と判定基準

## 目的

公式プレイヤーへ大量の実験アクセスを行わず、同じ公式クライアントコードに合成コメントを反復入力できる決定論的な実験環境を作る。第一段階では「匿名セッションのレスポンスを保存できる」「未記録通信を外へ出さず監査できる」ことまでを対象とする。

## 採用した境界

ブラウザーが見たHTTP交換を保存し、オフラインChromeの `Fetch` ドメインで同じURL、HTTPメソッド、リクエスト本文SHA-256に一致する応答だけを返す。ローカルHTMLへURLを書き換える方式は、CORS、CSP、相対URL、worker originを変えて公式コードの挙動に別の差を入れるため採用しない。アドレスバー上のoriginは元のURLのまま保ち、通信だけを記録応答に差し替える。

外部通信遮断は次の多層構成とする。

1. ページターゲットの全リクエストを捕捉し、ブラウザー生成のCORSプリフライトだけを監査付きでローカル合成し、それ以外のアーカイブ不一致を `BlockedByClient` にする。
2. ChromeのHTTP/HTTPSプロキシを待受のない `127.0.0.1:9` に固定する。
3. ホスト名を `~NOTFOUND` へ写像する。
4. QUICと非プロキシWebRTC UDPを無効化する。
5. background networking、component update、syncなどのブラウザー通信を無効化する。

第1層は再現応答の提供とページ通信の監査、第2〜5層はCDP対象外のブラウザープロセス通信に対する防護である。

## データ最小化

- キャプチャは専用の空の一時プロファイルで行う。
- Cookie、Authorization、リクエスト本文の平文、`Set-Cookie` は保存しない。
- POST照合には本文のSHA-256だけを使う。
- 動画・音声のレスポンス本文は保存しない。
- レスポンス本文は内容ハッシュ名で重複排除し、マニフェストから参照する。
- キャプチャ全体は `.gitignore` 対象とし、配布物へ含めない。

匿名ページのHTMLやAPI応答には短寿命のキーが含まれる可能性がある。したがってキャプチャは秘密情報に準じてローカルだけで扱い、共有が必要な研究結果は生データではなく集計値、仮説、再現用の合成入力へ変換する。

## 第一段階の合格条件

- キャプチャが `manifest.json` とSHA-256検証可能な本文を生成する。
- 同一アーカイブの再生で本文ハッシュ不一致を検出する。
- 再生中に未記録リクエストを外部へフォールバックせず、`audit.json` へ残す。
- `blockedCount = 0` で、ページが `readyState = complete` に到達する。
- 公式コメント描画コードが読み込まれ、Canvasまたは対応workerの生成を確認できる。

最後の2項目を満たさない場合も隔離機構の失敗とは限らない。記録時間不足、キャッシュ依存、worker、WebSocket、動画メディア依存を監査結果から分離する。

## 第二段階の実験対象

ローカルコメント応答の置換後、最小ケースから次の独立変数を増やす。

| 軸     | 初期値   | 拡張値                         |
| ------ | -------- | ------------------------------ |
| 位置   | `naka`   | `ue`, `shita`                  |
| サイズ | `medium` | `small`, `big`                 |
| 行数   | 1        | 2, 3, 10以上                   |
| 同時数 | 1        | 2, lane数前後, 過密            |
| 時刻差 | 0ms      | 境界前後を二分探索             |
| 本文幅 | 短文     | 等幅段階、全角・半角混在       |
| 処理順 | `no`昇順 | 配列逆順、`no`逆順             |
| 装飾   | なし     | 色、フォント、透過相当コマンド |

各仮説の導出ケースと検証ケースを分ける。特定のコメントアート作品は回帰確認には使うが、規則の導出には使わない。未知の時刻差、本文幅、複数行数をホールドアウトし、一致しない仮説はCOへ実装しない。

## 現在の未解決事項

- 公式プレイヤーのコメント描画がmain frame、worker、workletのどこまで分散しているか。
- 初期HTMLと静的バンドルだけでコメント描画を起動できるか、動画メタデータや視聴セッションの代替が必要か。
- `nvComment`以外にコメントレイヤー構成へ影響するAPIがあるか。
- WebSocket通信がコメント表示の初期化に必須か。
- 同じキャプチャ内の短寿命値を使ったPOST本文が毎回決定論的に再生成されるか。

これらはオンライン側へ試行回数を増やして解くのではなく、単一キャプチャの `audit.json`、ロード済みスクリプト一覧、DOM、Canvas、worker生成ログから切り分ける。
